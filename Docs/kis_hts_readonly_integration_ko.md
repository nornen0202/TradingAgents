# 한국투자 HTS 읽기 전용 연동 방안

## 결론

로컬 PC에서 eFriend Plus가 실행 중이면 TradingAgents 작업 에이전트가 해당 창을 활성화하고 화면을 캡처해 현재가, 호가, 차트, 수급 패널처럼 화면에 보이는 정보를 확인할 수 있다. 2026-07-17 로컬 점검에서 실제 eFriend Plus 창과 종목 화면 캡처까지 확인했다.

다만 eFriend Plus의 Windows 접근성 트리는 창 제목 외의 세부 컨트롤과 값을 제공하지 않았다. 따라서 HTS 화면을 안정적인 구조화 데이터 API처럼 취급할 수 없고, 화면 좌표·OCR에 의존한 자동 수집은 해상도, 창 배치, 팝업, 로그인 상태에 따라 쉽게 깨질 수 있다.

## 권장 우선순위

1. **KIS Open API를 정본으로 사용한다.** 계좌 잔고, 미체결, 체결, 기간손익, 권리 이벤트, 현재가와 미시구조 데이터는 기존 `tradingagents/portfolio/kis.py`, `tradingagents/dataflows/intraday/kis_quotes.py`, `tradingagents/dataflows/intraday/microstructure.py` 경로를 사용한다.
2. **HTS 화면은 읽기 전용 보조 확인에만 사용한다.** API에 없는 화면 전용 경보, 사용자가 지정한 특정 패널, API와 화면 값의 불일치 확인에 한정한다.
3. **파일 내보내기가 있으면 OCR보다 우선한다.** HTS가 CSV/Excel 내보내기를 제공하는 화면은 사용자가 직접 내보낸 파일을 읽는 방식이 좌표 자동화보다 재현성과 감사 가능성이 높다.
4. **주문은 자동화하지 않는다.** 매수·매도 버튼, 주문 수량·가격 입력, 정정·취소, 최종 제출은 금융 거래이므로 별도의 명시적 요청과 제출 직전 사용자 확인 없이는 수행하지 않는다.

## 안전한 수집 계약

- 기본 모드는 `READ_ONLY`다.
- 캡처 결과에서 계좌번호, 사용자 식별자, 인증 값과 주문번호는 저장·Pages 게시·Telegram 전송 대상에서 제외한다.
- 구조화 값은 화면 이름, 종목, 관측 시각, 관측 값, OCR 신뢰도, 원본 캡처 해시를 함께 기록한다.
- OCR 값만으로 `READY_NOW` 주문 상태를 만들지 않는다. KIS API 또는 사용자의 직접 확인이 있어야 한다.
- HTS와 API 값이 충돌하면 자동으로 어느 한쪽을 선택하지 않고 `NEEDS_MANUAL_RECHECK`로 표시한다.
- 로그인, 보안매체, 인증서, 비밀번호 입력 화면은 캡처·자동 입력 대상에서 제외한다.

## 단계별 도입안

### 1단계: API 정본 강화

- 기존 KIS 잔고·미체결·손익·현재가 수집의 성공/실패 receipt를 통합 전략 산출물에 연결한다.
- 필드별 `observed_at`, provider, freshness를 기록한다.
- 미국 종목의 execution-grade 데이터 한계는 별도 blocker로 유지한다.

### 2단계: 수동 요청형 HTS 확인

- 사용자가 확인할 화면과 종목을 지정했을 때만 창을 활성화한다.
- 화면 캡처 후 공개 시장 데이터만 요약하고 계좌 식별 정보는 즉시 마스킹한다.
- 읽은 값은 일회성 보조 근거로 표시하며 자동 주문 조건으로 승격하지 않는다.

### 3단계: 제한된 화면 어댑터

- 반복 가치가 큰 화면 하나씩 고정 레이아웃 프로필과 OCR 영역을 정의한다.
- 종목코드·화면명·관측시각 일치 여부를 선행 검증한다.
- 테스트 캡처 모음으로 숫자 인식 정확도와 소수점·부호 오류를 측정한다.
- 창 크기나 레이아웃이 달라지면 fail closed하고 API 경로로 되돌아간다.

## 채택하지 않는 방안

- HTS 화면 전체를 상시 녹화하거나 모든 텍스트를 무차별 수집하지 않는다.
- 화면 좌표만으로 계좌 데이터를 정본화하지 않는다.
- 비밀번호·인증서·OTP를 저장하거나 자동 입력하지 않는다.
- 투자 전략 보고서 생성과 주문 제출을 하나의 무확인 자동 단계로 연결하지 않는다.
